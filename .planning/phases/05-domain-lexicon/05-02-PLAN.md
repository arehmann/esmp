    - Test: USES_TERM is created only for the primary source class and classes that DEPENDS_ON that class (not test classes, not all classes containing the word)
    - Test: DEFINES_RULE edges exist between classes matching business rule naming patterns (Validator, Rule, Policy, Constraint, Calculator, Strategy) and their extracted terms
    - Test: DEFINES_RULE also created for classes with validation annotations (javax.validation.Constraint or jakarta.validation.Constraint)
    - Test: Re-extraction produces same edge count (idempotent MERGE)
    - Test: Curated term definitions survive re-extraction (verify curated=true term's definition is unchanged after re-linking)

  <action>
    1. Add two new linking methods to `LinkingService`:

       `linkBusinessTermUsages(ExtractionAccumulator acc)`:
       - For each BusinessTermData in acc.getBusinessTerms(): create USES_TERM edge from the primary source class to the BusinessTerm node
       - Also create USES_TERM edges from classes that DEPENDS_ON the primary source class
       - Exclude test classes (sourceFilePath contains '/test/')
       - Cypher pattern:
         ```
         MATCH (c:JavaClass {fullyQualifiedName: $classFqn})
         MATCH (t:BusinessTerm {termId: $termId})
         MERGE (c)-[r:USES_TERM]->(t)
         RETURN count(r) AS cnt
         ```
       - For dependent classes:
         ```
         MATCH (c:JavaClass {fullyQualifiedName: $primaryFqn})<-[:DEPENDS_ON]-(dep:JavaClass)
         WHERE NOT dep.sourceFilePath CONTAINS '/test/'
         MATCH (t:BusinessTerm {termId: $termId})
         MERGE (dep)-[r:USES_TERM]->(t)
         RETURN count(r) AS cnt
         ```

       `linkBusinessRules(ExtractionAccumulator acc)`:
       - Query for all JavaClass nodes whose simpleName matches business rule pattern: `simpleName =~ '.*(Validator|Rule|Policy|Constraint|Calculator|Strategy).*'`
       - For each matched class, extract terms from its simpleName using same camelCase split logic
       - Create DEFINES_RULE edge from the class to each extracted BusinessTerm
       - Also detect classes with validation annotations (HAS_ANNOTATION edges to javax/jakarta.validation.Constraint)
       - Cypher:
         ```
         MATCH (c:JavaClass)
         WHERE c.simpleName =~ '.*(Validator|Rule|Policy|Constraint|Calculator|Strategy).*'
            OR EXISTS {
              MATCH (c)-[:HAS_ANNOTATION]->(a:JavaAnnotation)
              WHERE a.fullyQualifiedName IN ['javax.validation.Constraint', 'jakarta.validation.Constraint']
            }
         MATCH (t:BusinessTerm)
         WHERE t.termId IN $termIds
         MERGE (c)-[r:DEFINES_RULE]->(t)
         RETURN count(r) AS cnt
         ```
         Where $termIds is derived from splitting the class simpleName.

    2. Wire into `linkAllRelationships()`:
       - Add calls to `linkBusinessTermUsages(acc)` and `linkBusinessRules(acc)` after existing linking methods
       - Update `LinkingResult` record: add `int usesTermCount` and `int definesRuleCount` fields
       - Update the log statement and return to include new counts

       - Test USES_TERM edge creation
       - Test DEFINES_RULE edge creation for Validator-named class
       - Test idempotency (run linking twice, same edge counts)
       - Test curated protection (set curated=true on a term, re-extract, verify definition unchanged)

Looking at the source: it has a no-arg constructor that hardcodes queries. So `LexiconValidationQueryRegistry` should NOT extend it. Instead, since ValidationService uses `List<ValidationQueryRegistry>`, we need to either:
