Create `com.esmp.indexing.api.IndexingController` as a `@RestController` with `@RequestMapping("/api/indexing")`.

**Endpoint: `POST /api/indexing/incremental`**
- Accepts `@RequestBody IncrementalIndexRequest request`
- Validates: if both `changedFiles` and `deletedFiles` are null or empty AND `sourceRoot` is provided, treat as a full re-index (pass all files under sourceRoot as changedFiles). This implements the "single unified endpoint handles both incremental and full module re-index" locked decision.
- If sourceRoot is blank/null: return 400 with error message.
- Calls `incrementalIndexingService.runIncremental(request)` synchronously (locked decision: synchronous, blocks until complete).
- Returns `IncrementalIndexResponse` as JSON with 200 status.
- If errors list in response is non-empty, still return 200 (partial success) — the errors field communicates individual file failures.

**Full re-index path:** When changedFiles and deletedFiles are both empty but sourceRoot is provided, scan sourceRoot for all .java files (using the same `Files.walk().filter(*.java)` pattern from ExtractionService), set them as changedFiles, and pass to runIncremental. This enables the "full module re-index (sourceRoot only)" use case per the locked decision.
