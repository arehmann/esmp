    TWO independent fixes in this task:

    FIX 1 — HAS_ANNOTATION FQN mismatch:
    The root cause is that ClassMetadataVisitor.resolveAnnotationName() returns simple names ("Entity")
    when type resolution fails, but JpaPatternVisitor.resolveAnnotationFqn() maps them to FQNs
    ("javax.persistence.Entity") and stores the JavaAnnotation node with that FQN.

    The fix: Normalize annotation names to FQNs in ClassMetadataVisitor.resolveAnnotationName() by
    adding the SAME simple-name-to-FQN fallback switch that JpaPatternVisitor already uses. Extend
    the fallback switch to cover ALL common annotation simple names:
    - "Entity" -> "javax.persistence.Entity"
    - "Table" -> "javax.persistence.Table"
    - "Service" -> "org.springframework.stereotype.Service"
    - "Repository" -> "org.springframework.stereotype.Repository"
    - "Controller" -> "org.springframework.stereotype.Controller"
    - "RestController" -> "org.springframework.web.bind.annotation.RestController"
    - "Component" -> "org.springframework.stereotype.Component"
    - "Autowired" -> "org.springframework.beans.factory.annotation.Autowired"
    - "Inject" -> "javax.inject.Inject"
    - "Query" -> "org.springframework.data.jpa.repository.Query"

    This ensures c.annotations stores FQNs that match JavaAnnotation.fullyQualifiedName, so the
    existing linkAnnotations() Cypher will work without changes.

    IMPORTANT: This fix also benefits the stereotype detection from Plan 05 — if Plan 05 runs first,
    the simple name fallback in the sets handles it. If Plan 06 runs first, the FQN normalization
    in resolveAnnotationName handles it. Either way, both fixes together are belt-and-suspenders.

    FIX 2 — QUERIES repository-to-entity resolution:
    linkQueryMethods() looks up qm.declaringClassFqn() (the repository FQN) in tableMappings
    (keyed by entity FQN). It needs to first resolve the repository to its entity, then look up
    the entity in tableMappings.

    The fix: In linkQueryMethods(), before the main loop, build a repositoryToEntity map by
    querying Neo4j for all JavaClass nodes that implement a JpaRepository/CrudRepository interface.
    Use a Cypher query to extract the class node's implementedInterfaces and resolve generic type
    parameters. Since we cannot resolve generics from Cypher, use a simpler approach:

    Add a new method to ExtractionAccumulator or use a new data structure. Actually, the simplest
    approach is: for each QueryMethodRecord, look up the declaring class in acc.getClasses(), read
    its implementedInterfaces list, check if any interface is a known repository super-type
    (JpaRepository, CrudRepository, etc.), and if so, iterate acc.getTableMappings() to find which
    entity class the repository manages.

    Better approach: The repository interface EXTENDS JpaRepository<SampleEntity, Long>. The
    ClassMetadataVisitor already captures superClass for extends. For interfaces, the
    implementedInterfaces list will contain the parent interface FQN. However, we cannot get
    the generic type parameter from the FQN string alone.

    SIMPLEST approach: Use a Cypher query in linkQueryMethods() that traverses
    the IMPLEMENTS relationship from the repository to find the entity it manages. Since the
    repository extends JpaRepository<Entity, ID>, and the entity class has a MAPS_TO_TABLE edge
    to its table, we can do:

    ```cypher
    MATCH (m:JavaMethod {methodId: $methodId})
    MATCH (repo:JavaClass {fullyQualifiedName: $repoFqn})
    MATCH (repo)-[:DEPENDS_ON|IMPLEMENTS*1..3]->(entity:JavaClass)-[:MAPS_TO_TABLE]->(t:DBTable)
    MERGE (m)-[r:QUERIES]->(t)
    RETURN count(r) AS cnt
    ```

    This traverses from the repository class through any chain of DEPENDS_ON or IMPLEMENTS edges
    (up to 3 hops) to find an entity class that has a MAPS_TO_TABLE edge to a DBTable. This
    is a graph-native solution that doesn't need to parse generic type parameters.

    NOTE: The current code iterates acc.getQueryMethods() and does map lookups. Replace the
    tableMappings lookup with the Cypher graph traversal above. Keep the iteration pattern but
    change the inner Cypher to include the graph traversal.

    Update LinkingServiceIntegrationTest to add test cases for both fixes:
    - HAS_ANNOTATION: Create a JavaClass node with annotations=["javax.persistence.Entity"] and
      a JavaAnnotation node with that FQN, verify HAS_ANNOTATION edge is created.
    - QUERIES: Create a JavaMethod on a repository class, create the repository -> entity
      IMPLEMENTS chain and entity -> table MAPS_TO_TABLE edge, verify QUERIES edge is created.
