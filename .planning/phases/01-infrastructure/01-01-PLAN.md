    Create `src/main/java/com/esmp/infrastructure/config/QdrantConfig.java`:
    - `@Configuration` class
    - `@Bean` method creating `QdrantClient` using `QdrantGrpcClient` builder
    - Read host and port from `@Value("${qdrant.host}")` and `@Value("${qdrant.port}")`
    - Use the gRPC channel builder pattern from the Qdrant Java client

    Create `src/main/java/com/esmp/infrastructure/health/QdrantHealthIndicator.java`:
    - Implements `HealthIndicator` (from `org.springframework.boot.actuate.health`)
    - `@Component` annotated
    - Injects `QdrantClient` via constructor
    - `health()` method calls `qdrantClient.healthCheckAsync().get(3, TimeUnit.SECONDS)`
    - Returns `Health.up()` with detail "qdrant: reachable" on success
    - Returns `Health.down()` with detail "qdrant: unreachable" and exception on failure
    - Follow exact pattern from RESEARCH.md Pattern 2

    Create `src/main/java/com/esmp/infrastructure/startup/DataStoreStartupValidator.java`:
    - `@Component` class implementing `ApplicationRunner` (from `org.springframework.boot.ApplicationRunner`)
    - Per CONTEXT.md locked decision: "startup fails fast if any store unreachable"
    - Injects `List<HealthIndicator>` via constructor (Spring auto-collects all HealthIndicator beans — this includes the built-in Neo4j and DataSource indicators plus our custom QdrantHealthIndicator)
    - `run(ApplicationArguments args)` method iterates over all health indicators:
      1. For each `HealthIndicator`, call `health()` and check the status
      2. If ANY indicator returns `Status.DOWN`, log the failing component name and throw `IllegalStateException("Data store startup validation failed: {component} is DOWN — {details}")`
      3. This causes Spring Boot to abort startup immediately
    - Use SLF4J logger to log each check: `"Startup validation: checking {indicator class name}..."` and `"Startup validation: {indicator class name} is UP"`
    - On success, log: `"All data store connections verified — startup validation passed"`
    - IMPORTANT: The `ApplicationRunner.run()` method executes AFTER the application context is fully initialized but BEFORE the app is considered "ready". Throwing from `run()` causes `SpringApplication.run()` to fail and the JVM to exit — this is the correct fail-fast behavior.
    - NOTE: HikariCP (MySQL connection pool) already fails fast if MySQL is unreachable due to its connection-timeout default (30s). Spring Data Neo4j's auto-configured driver also fails at context initialization if Neo4j is unreachable. The DataStoreStartupValidator provides a UNIFIED safety net that catches ALL stores including Qdrant (which has no auto-configured fail-fast). This ensures consistent behavior regardless of individual driver defaults.

    Create `src/main/resources/application.yml` — base configuration exactly from RESEARCH.md:
    - spring.application.name: esmp
    - spring.threads.virtual.enabled: true
    - spring.datasource: MySQL on localhost:3306/esmp with env var password
    - spring.jpa.hibernate.ddl-auto: none (CRITICAL — Flyway owns schema)
    - spring.flyway: enabled, classpath:db/migration
    - spring.neo4j: bolt://localhost:7687 with env var password
    - management.endpoints: expose health, info, metrics, prometheus; show-details always; probes enabled
    - qdrant.host: localhost, qdrant.port: 6333
    - server.port: 8080
